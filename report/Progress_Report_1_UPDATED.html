1. App / Game Prototype Generation

1.1 Concept & Idea

BAN's Protocol: Adaptive Horde Survival is a top-down survival shooter set inside a randomly generated maze. The enemy horde is not driven by random spawns or a fixed difficulty curve — a fuzzy-logic "AI director" continuously reads the player's live state and decides how aggressive the horde should be and how many enemies to send.

The defining idea added since the last report is stealth-aware difficulty: the director now cares not only about how strong the player is, but about whether the horde actually knows where the player is. A player who hides in fog and stays quiet is left alone almost regardless of how well-armed they are; a player who is seen, loud and loaded gets the full swarm. Difficulty has become a resource the player manages through positioning and noise discipline, not just through health and ammo.

1.2 Problem Statement

Conventional games scale difficulty with crisp thresholds or fixed timers (e.g. "after 60 seconds, spawn 10 enemies"). These rules are brittle: they snap between levels, ignore the player's actual situation, and cause either boredom (too easy) or a frustrating death spiral (piling enemies onto an already-weak player). The problem we address is dynamic difficulty balancing under multiple, overlapping, vague conditions: how can a game smoothly and believably adapt its challenge to the player's moment-to-moment condition — health, firepower, noise, how swarmed they already are, and how exposed they are — when every one of those signals is naturally fuzzy ("fairly low health", "kind of loud", "mostly hidden")?

1.3 Target Users

  • Casual and mid-core action / survival players who want a challenge that always feels fair.
  • Game designers and students studying adaptive / dynamic-difficulty AI directors.
  • Educators and learners using the live on-screen fuzzy visualisation as an interactive demonstration of Mamdani fuzzy inference.

The prototype is a complete, playable build in pure HTML5 Canvas + JavaScript. Every control and readout is drawn on the canvas (no HTML form widgets). The interaction flow is:

  • Gameplay: WASD/Arrows move, mouse aims, Click/Space shoots, R reloads, Q / 1 / 2 / 3 switch weapons, G toggles god mode, B / V toggle HUD panels, Esc pauses.

1.6 Rationale — Why Fuzzy Logic Is Appropriate

The director's decision depends on human, imprecise concepts: "a bit low on health", "fairly loud", "almost out of ammo", "kind of swarmed", "barely spotted". These categories overlap and have no sharp boundary — exactly the uncertainty fuzzy logic is designed for. Concretely:

  • Smooth blending — overlapping membership functions interpolate between rules, so difficulty ramps and eases gradually instead of snapping between levels.
  • Natural rule authoring — rules read in plain language ("IF health is Low AND ammo is Depleted THEN threat is Passive").
  • Multi-input fusion — five uncertain inputs are combined into one coherent decision. With five variables, hand-coding every combination is infeasible; fuzzy inference fuses them cleanly (and we generate the full 729-rule grid from three transparent scoring policies — see §2.4).
  • Believable behaviour — the resulting difficulty curve feels reactive and intentional rather than scripted, and the stealth dimension (Exposure) gives the player real agency over the challenge.

2.4 Expected Fuzzy Rules — the three generating policies

The inference is a Mamdani model with AND = minimum, aggregation = maximum, defuzzified by centroid (with a parallel Sugeno path, §3.8). Rather than hand-write 729 rules, the engine generates the complete 3⁶ = 729-rule grid; each rule carries three consequents (Threat, Supply, Composition), each assigned by its own transparent policy over the ordinal term levels (0/1/2):

THREAT  = 0.55·(health+ammo) + 0.8·noise − 0.6·pressure + 1.7·exposure + 0.6·(skill−1)
          ≤ 2.3 → Passive · ≤ 4.6 → Tactical · else → Overwhelming

SUPPLY  = (2−health) + (2−ammo) + 0.6·pressure                      (help the desperate)
          ≤ 1.6 → Scarce · ≤ 3.0 → Balanced · else → Generous

COMPO   = 0.5·(health+ammo) + 0.7·noise + 0.8·exposure + 0.5·skill  (heavier when strong/loud/seen/skilled)
          ≤ 2.0 → Swarm · ≤ 3.8 → Mixed · else → Heavy

weight  w = 0.6 (at a band boundary) … 1.0 (deep inside a band)
  • Capability raises Threat but is damped (0.55) — strong, loaded players get pushed.
  • Noise raises Threat (0.8) — being loud is punished.
  • Pressure lowers Threat (−0.6) — already-swarmed players get relief, not a pile-on.
  • Exposure dominates Threat (1.7) — a Hidden & Quiet player stays Passive no matter how strong.
  • Skill is centred (0.6·(skill−1)): Average adds 0, so an average player sees exactly the pre-Phase-3 balance; Novice is eased and Expert is pushed — adaptive difficulty without breaking the tuning.
  • Supply ignores Noise/Exposure so it is genuinely decoupled from Threat; Composition rewards seen/loud/skilled play with a tougher enemy mix.

Rule-base composition (machine-enumerated over all 729 rules): Threat = 271 Passive · 308 Tactical · 150 Overwhelming; Supply = 189 Scarce · 270 Balanced · 270 Generous; Composition = 156 Swarm · 408 Mixed · 165 Heavy. The Threat distribution is deliberately mercy-biased — well under a third of situations unleash the full horde, reserved for players who are healthy, loaded, loud, seen and skilled.

(a) How many inputs and outputs are actually used, and why each one. The controller is a six-input, three-output MIMO fuzzy model. It grew MISO (1 output) → MIMO (2, then 3) across the phases. Every variable earns its place — none is decorative:

Why three separate outputs and not one. The three decisions are genuinely independent: a hidden but hurt player should face low Threat yet receive generous Supply; a skilled, spotted player warrants high Threat and a Heavy composition. Folding them into one number could not express these — so the same antecedents drive three consequent columns, each with its own scoring policy (§2.4). (Score, Kills and Wave are crisp outcome metrics, not fuzzy outputs — see §3.9.)

(b) Why 729 rules. A complete grid is termsinputs. With six inputs of three terms, the full antecedent space is 3⁶ = 729, and the engine generates all of it so no reachable game state is ever unhandled — the number is a direct, unavoidable consequence of choosing six meaningful inputs, not padding. Capping lower would silently drop inputs that define the game's feel (Pressure, Exposure, Skill):

The brief's "at least 2 rules" is a floor, not a target; 729 is the natural, complete consequence of choosing six inputs and demanding total coverage. Crucially the rules are not hand-written (which would invite contradictions at this scale) — all 729 are generated from three transparent scoring policies (§2.4), so the grid stays consistent and, as this shows, the policy approach scales to any number of inputs without extra authoring effort — a key engineering advantage over hand-built rule bases.

Managing 729 rules for a human. Because the base is generated and every rule fires with a weight, the player never confronts 729 rules at once — typically only a handful fire meaningfully at any instant. The in-game ACTIVE FUZZY RULES panel lists the top-firing rules live, and the Fuzzy Analytics screen (§3.10) shows a rule-firing heatmap so the whole 729-cell base can be inspected at a glance.

(c) Direct rules or combined rules? A direct rule has a single antecedent — one input mapped straight to the output (e.g. the lecture example "IF x is A1 THEN z is C3"). A combined rule joins several antecedents with a fuzzy operator AND/OR (e.g. "IF x is A2 AND y is B2 THEN z is C2"). Every one of our 729 rules is a combined (conjunctive) rule: each rule fixes one term for all six inputs, joined by AND = min. This is deliberate — Threat depends on the interaction of the five signals (a strong player who is hidden must be treated completely differently from a strong player who is spotted), and that cross-coupling can only be expressed by combined rules. A set of independent direct rules (one per input) could never produce the "strong-but-hidden ⇒ Passive" behaviour that defines the game.

3.0 System Flow — How It All Fits Together

The game is a closed feedback loop ticking at 60 FPS. One pass of the loop (frame() in main.js → update() in mechanics.js) does this:

The key property is that there is no scripted timeline: difficulty is an emergent equilibrium between the player and the fuzzy director. Player actions change the six inputs → the engine re-decides the three outputs → the director reshapes the horde → the new horde changes the player's state → the loop repeats. Skill closes a second, slower loop: sustained good play raises the Skill input, which permanently nudges the whole system harder (and vice-versa). Clearing a boss triggers a sector reshuffle (new maze + modifier) and a perk choice, so the loop also has long-term roguelite progression layered on top.

3.2 The Fuzzy Pipeline (Step by Step)

Every inference pass follows the classic Mamdani sequence. Each step corresponds to real code.

Step 1 — Sampling & Normalisation. Every 10 frames the loop reads five raw values and normalises each to 0–100:

  • Health = HP / maxHP × 100
  • Ammo = (magazine + reserve) / total capacity × 100
  • Noise = a running meter (+12…32 per shot by weapon, +0.5/frame while moving, −0.35/frame decay)
  • Pressure = (100 − nearestDist/5) + nearbyCount·2, clamped 0–100
  • Exposure = the detection meter G.detect (LOS up, fog down, loud-noise floor)

Step 2 — Fuzzification. Each crisp value passes through its term functions (tri, trapL, trapR) to give a membership 0–1. Overlap lets the system blend between rules instead of snapping.

Step 3 — Rule evaluation. Each of the 729 rules combines its six antecedents with AND = min, then multiplies by the designer weight w: strength = min(μ₁, …, μ₆) · w. Firing rules (strength > 0.001) are recorded for the on-screen panel.

Step 4 — Aggregation. Rules pointing at the same output term are combined with OR = max, giving one aggregated strength per output set (Passive / Tactical / Overwhelming).

Step 5 — Defuzzification. Centroid over the clipped, unioned output sets, sampled in steps of 2:

for x = 0..100 step 2:
    μ = max over terms of min(agg[term], term_MF(x))
    num += x·μ ; den += μ
threat = num / den

Centroid gives smooth, continuous output — small state changes move Threat gently, exactly what a dynamic-difficulty director needs.

The crisp Threat (and its band) drives the director:

ParameterHow Threat affects it
Horde size per wavecount = round((0.5 + aggr·4.5) · Density) — higher Threat sends more enemies (enemy cap 40).
Spawn cadencespawnTimer = (240 − Threat·1.5) / Pace — a strong, seen player faces wave after wave.

Stealth override: when detection < 15 the director ignores Threat entirely and spawns purely from Noise — dead silence yields at most a lone wanderer; gunfire scales the swarm with how loud you are.

3.4 The Stealth Loop — Closing It with Exposure

Noise and Exposure together give the player agency over difficulty. Firing is loud and draws line-of-sight, pushing Exposure toward Spotted, which fires high-threat rules. Slipping into fog drains detection ~5× faster (hiding resets aggression in about a second) and holding fire lets Noise decay to Quiet, dropping Threat. Difficulty becomes a strategic resource — cleanly expressible only because Noise and Exposure are fuzzy quantities, not hard on/off flags.

3.5 Live Visualisation of the Fuzzy System

So the inference is not a black box, the game renders it in real time: the Threat gauge (needle + band arc), five MF graphs (one per input, moving marker + dominant-term highlight), and the Active Rules panel (which rules fire and at what strength, as percentage bars) — an X-ray of the inference each instant. A minimap shows walls, fog, the player and enemies so the stealth state is legible too.

3.8 Fuzzy Modelling — Takagi–Sugeno Inference, Step by Step

The same four-step process can be run in Takagi–Sugeno (TSK) style. As the lecture notes, Sugeno inference is identical to Mamdani except for the rule consequent: instead of a fuzzy set, each rule outputs a crisp value — a constant singleton k (zero-order) or a function f(x,y) of the inputs (first-order). We run the very same §3.7 state (Health 50, Ammo 55, Noise 55, Pressure 15, Exposure 70).

STEP 1 — Fuzzification (identical to Mamdani)

Unchanged from §3.7: the crisp inputs give Med 1.0, Adeq 1.0, Mod 1.0, Safe 1.0, and Exposure Suspicious 0.40 / Spotted 0.20.

STEP 2 — Rule Evaluation (firing strength w, then a singleton spike — no clipping)

The antecedent truth value (firing strength w) is found exactly as before with AND = min × weight. But the consequent is now a singleton spike at a crisp output k, scaled to height w — there is no fuzzy set to clip. 3.9 The Game's Final Output — the Player's End Goal (Score · Kills · Wave)

It is important to separate two different kinds of "output":

  • The fuzzy controller's output is a single value — Threat / Aggression (§2.5, §3.7–3.8). Its job is to set the difficulty, not to score the player.
  • The game's final output — what the player is actually trying to maximise — is the run result shown on the OVERRUN (game-over) screen: Score · Kills · Wave (drawn in js/screens.js). This is the endgame: survive the fuzzy-scaled horde for as long as possible and rack up the highest Score / Kills / Wave.

How the three are produced in js/mechanics.js:

Outcome metricHow it is earned
KillsG.kills++ on every enemy destroyed.
ScoreG.score += e.pts — points by archetype: grunt 10, runner 15, brute 25 (tougher enemies are worth more).
Waveif (G.kills % 15 === 0) G.wave++ — the wave counter rises one step every 15 kills, so it tracks how deep the run went.

Because the fuzzy director keeps the challenge fair, a skilled player survives longer, which pushes all three metrics up together. To make a run easy to read at a glance, each metric can be banded Low / Medium / High (indicative targets for a default Pace 0.6 / Density 1.0 run):

3.10 Fuzzy Analytics — an In-Game Inspection Tool (Phase 3)

Pressing C in play opens a dedicated Fuzzy Analytics overlay that freezes the game and exposes the engine's internals — a teaching/verification instrument built directly on the live 729-rule controller:

  • Rule-firing heat-map — all 729 rules as a 27×27 grid, each cell brightening with how often that rule has fired this run, so the whole base is inspectable at a glance.
  • Defuzzifier comparison — the same aggregated Threat set defuzzified four ways side by side: centroid, bisector, mean-of-maxima and Sugeno weighted average — with a runtime Mamdani ↔ Sugeno toggle that switches which method actually drives the game.
  • "Why this threat?" explainability — one plain-English sentence generated from the dominant firing rule (also shown live under the rules panel).
  • Fuzzy weapon advisor — a small separate FIS (nearest-enemy distance × crowd size → weapon) that hints which weapon suits the moment; fuzzy decision-support, not just difficulty.
